2 Use of the Database

2.1 Introduction

This chapter provides an overview of how to perform simple operations using SQL in TOTEM. In the examples that follow, we assume that you have created a database named mydb and have started tsql.

TOTEM is a database management system based on the object-agent model. The concept of storing data in tables has become almost inherently common knowledge It is one of the most basic concepts of a relational database; however, there are other ways to organize a database. Files and directories on Unix-like operating systems are examples of hierarchical databases. A more modern development is the object-oriented database. Compared to the traditional object-oriented The object-agent model is more flexible, more flexible, and easier to use than the traditional object-oriented model.

2.2 Basic concept of Proxy

The so-called "proxy" refers to the process of switching all or some properties of an object (source object), such as name change, operation, etc., to derive another object (proxy object). The process of deriving another object (proxy object) by performing certain switching operations on all or some of the properties of an object (source object), such as name change, arithmetic, and so on. Through object proxies, the user can split and combine different objects to form new views. The object proxy allows the user to combine different objects into a new view of the object. Compared with the original object-oriented model, the object proxy model is more flexible, operation It is more flexible and easier to use.

Both source and proxy objects belong to their own classes. Thus, the concept of source and proxy classes is formed. Proxy objects with switching operations and modes are gathered in a proxy class. There is a difference between a proxy class "proxying" the source class and "inheriting" in the general sense. A proxy class is more like a view of the source class. The proxy class is more like a view of the source class, with different proxy objects representing different roles played by the source object. Multiple proxy classes can be defined at the same time on a source class. Proxy Classes It is possible to proxy on several source classes at the same time by concatenation or merge operations. It is also possible to define additional proxy classes on a proxy class. It can also define other proxy classes on the proxy class. Proxy relationships are in the shape of a directed graph. The classes at the top of the proxy hierarchy are called base classes, and they can be considered are proxy classes without source classes.

In the storage model, the objects of the basic class occupy the storage space and store the actual data. Most of the data in the database is stored in the base class. Proxy properties (called "dummy properties") in proxy objects only have schema and no actual physical data elements. Between the source object and the proxy object are linked by bidirectional pointers, and a switch operation determines the value of the dummy property in the proxy object. Proxy objects can extend their own properties The proxy object can extend its own properties, but it is these extended properties that actually occupy the storage space.

The definition of a proxy class is somewhat similar to the definition of a view in a relational database. For general query operations, the proxy class is exactly the same as a normal class. The proxy class is exactly the same as a normal class when it comes to general query operations.

2.3 Creating a basic class

The creation of a base class or proxy class in TOTEM corresponds to the creation of a table in traditional relational data. Basic classes are classes without source classes, they are the most basic classes and all their properties are real properties. It is essentially the same as a relational table, but it has methods. So we define a set of commands for the basic class, which are similar to those for defining a basic table in a relational database, including: defining the class name, attribute name and attribute type; establishing constraints, such as whether the attribute is allowed to be null, the default value of the attribute, etc.; creating a method, defining the method name, parameter type, return type, and and function body.

For example, the SQL statement to define a basic class (which stores information about all faculty members) faculty is as follows:

CREATE CLASS faculty
(
Id INT NOT NULL UNIQUE,
Name VARCHAR(20) NOT NULL,
birth INT,
department VARCHAR(30),
salary INT
);

2.4 Creating a proxy class

Defining a proxy class requires a proxy rule (essentially a query statement) to define the formal transformations to be performed on the source class. In order to ensure the clarity of the proxy relationship, the proxy rules must be somewhat qualified. In the object proxy model, only four simple proxy rules，which are Select, Join, Union, and Group proxies.

The proxy rules can be defined with certain semantic constraints, so that the object being proxied meets certain conditions. When an object is updated, the system automatically maintains the proxy rules according to these conditions. When an object is updated, the system automatically maintains consistency between objects based on these conditions. If no semantic constraints are declared, then a loose proxy class is defined (see the rest of this section) in which the objects are maintained manually by the user.

2.4.1 Define Select proxy class

Select type proxy allows to select only the source object from a class to be proxied. The source object and the proxy object are one-to-one correspondence. A select proxy class for the faculty class can be created by the following statement.

CREATE SELECTDEPUTYCLASS computer_faculty (field VARCHAR(30))
AS
(SELECT id, name, (extract(year from current_date) - birth) AS age FROM faculty
WHERE department ='computer')
WRITE (faculty.birth = 'extract(year from current_date)－age');

The computer_faculty proxy class directly inherits the id, name attributes from the source class faculty, and adds an attribute field to represent the faculty's area of expertise. The value of the birth attribute is converted to age by the expression (extract(year from current_date)-birth) AS age. The constraint WHERE department = 'computer' declares that the proxy class only proxies objects in the faculty class that belong to the computer department. The WRITE clause defines a write-switch operation for the age dummy attribute. It allows the age attribute to be updated.

2.4.2 Defining the Join proxy class

Join type proxy is a proxy on the result of the connection of several classes. It can achieve the effect of "multiple inheritance" like presentation. A proxy object in each source class corresponds to a source object.

Example: Create a proxy class assistant to store information about all students (both students and teachers).

CREATE JOINDEPUTYCLASS assistant
AS
( SELECT student.name, salary FROM student,faculty
WHERE student.id = faculty.id );

2.4.3 Defining the Union proxy class

Union proxy is equivalent to combining the results of several select proxies in a single proxy class.

Example: Create a union proxy class person for students and faculty, abstracting the two types of objects into one object of the same pattern.

CREATE UNIONDEPUTYCLASS person
AS
( SELECT name, birth FROM student UNION
SELECT name, birth FROM faculty);
select * from person;

2.4.5 Defining loose proxy classes

The proxy classes defined in the previous sections all contain semantic constraints and are called strict proxy classes. Once a strict proxy class is defined, the set of objects in it is automatically determined by the system based on the constraints and cannot be arbitrarily inserted or removed by the user. A proxy class that does not contain semantic constraints in the proxy rules is called a loose proxy class. After definition, the set of objects in the loose proxy class is maintained manually by the user. The difference between the proxy rules of a loose proxy class and a strict proxy class is that:

  1. IMPRECISE keyword must be written;

  2. no semantic constraints can be declared;

An example of creating a non-strict select proxy class for the student class is given below. This class is used to store information about "my friends". It is up to the user to determine which objects in the student class can be "my friends".

CREATE IMPRECISE SELECTDEPUTYCLASS my_friends(telephone int)
AS ( SELECT name FROM student);

2.5 Adding objects to a class

TOTEM appends objects to a class in two cases. (1) creating new objects in the base class. (2) Appending objects to a loose proxy class

2.5.1 Appending objects to basic classes

The INSERT command is used to create a new object in the basic class with the following syntax structure:

INSERT INTO class [ ( column [, ...] ) ]
{ DEFAULT VALUES | VALUES ( expression [, ...] ) | SELECT query }

The INSERT statement is entered as

  1. class, the name of the existing base class.
  2. COLUMN, the name of the field in the class.
  3. DEFAULT VALUES, where all fields are populated with NULL or the value declared with the DEFAULT clause at creation.
  4. expression , a valid expression or value given to the column.
  5. QUERY, a valid query.

The output of the INSERT statement is.

  1. INSERT oid 1, if only one object is inserted, returns this information, where oid is the numeric identifier of the object being inserted.
  2. INSERT 0 # returns this information if multiple objects were inserted, where # is the number of objects inserted.

INSERT allows us to insert new objects into the class. We can insert multiple objects at once, and the columns/fields in the target list can be in any order. Columns/fields that do not appear in the target column will be inserted with default values: DEFAULT values defined or NULL. If a NULL value is inserted into a column defined as NOT NULL, TOTEM will reject the new object. If the expression is not of the correct data type, an attempt will be made to perform automatic type conversion. To insert data into a class, you must have insert permission, and also have select permission to handle any classes declared in the WHERE clause. Note that the "columns/fields" mentioned here correspond to the properties of the object.

An example of the first INSERT usage: insert an object into the films. The command is as follows:

INSERT INTO films VALUES
('UA502','Bananas',105,'1971-07-13','Comedy',INTERVAL '82 minute');

The second usage example also inserts an object into the films, which omits the field len, so the default value of len will be stored as NULL. The command is as follows:

INSERT INTO films (code, title, did, date_prod, kind)
VALUES ('T_601', 'Yojimbo', 106, DATE '1961-06-16', 'Drama');

The user can also insert an object from another class tmp into the films with the following command:

INSERT INTO films SELECT * FROM tmp;

2.5.2 Appending objects to loose proxy classes

Append objects to the loose proxy class with ADD INTO, the syntax structure of the statement is as follows:

ADD ANY|INEXISTENT INTO name FROM source WHERE condition

The ADD INTO command can only be used for loose proxy classes. Loose proxy classes are created without any objects, so if the user wants to add some objects from the source class to the If you want to add some objects from the source class to the loose proxy class, you need to use this command. The ADD INTO syntax is explained as follows:

  1. ANY, regardless of whether the object selected in the where clause has already been proxied or not, adds it to the proxy class, so that one object in the proxy class corresponds to more than one proxy object.
  2. INEXISTENT, in contrast to ANY, if the source object selected in the where clause is found to have already existed in the proxy class for at least one
  3. name, the name of the inserted target class.
  4. source, the name of the source class to which the Source object belongs.
  5. condition, a Boolean expression used to select the source object, which may not contain subqueries.

The output of the ADD INTO statement is:

  1. COUNT, success, returns the number of joined objects.
  2. ERROR: can not insert into a precise deputy class, that is, TOTEM prohibits the use of ADD INTO to strictly proxy class Adding objects

2.6 Query

The query language of the object proxy database provides support for querying classes and proxy classes, and also provides cross-class querying based on path expressions. The query language of the object proxy database provides support for querying classes and proxy classes. For common queries (meaning queries that do not involve proxy classes and proxy objects), the query processing process of object proxy database can follow the common implementation methods of traditional databases are used. However, for the query processing involving the virtual properties of the proxy class, special processing is needed to make the proxy class queries can be described and implemented in the same form as the queries of traditional databases. TOTEM, an object proxy database system, has established a comprehensive query mechanism, which makes it possible for users to query classes and proxies freely and transparently, and also implements a special navigational query mechanism, i.e., cross-class queries. In the object proxy database, proxy relationships between (proxy) objects are represented by physical address pointer links.

The proxy relationships between (proxy) objects are represented by physical address pointer links. Therefore, the objects form an object hierarchy connected by proxy relationships. Using the object hierarchy, you can start from a (proxy) object of a (proxy) class and class, you can query the (proxy) objects of other (proxy) classes that are directly or indirectly related to it. Cross-class queries make use of a path of object pointers between (proxy) objects is formed, and a path expression is used to reflect the syntactic structure of the path.

2.6.1 General Query

A query is a procedure or command that retrieves data from a database, TOTEM uses the SELECT command to declare a query. The generic SELECT command syntax is:

SELECT select_list FROM class_expression [sort_specification]

The various parts of the SELECT syntax are described in detail next. The simplest form of the query is as follows:

SELECT * FROM class1;

Assuming there is a class called class1, this command will retrieve all properties of all objects from class1. Selecting a column declaration * means that all properties provided by the class expression. It is also possible to select a subset of the optional columns and even compute the properties before retrieving them. Suppose class1 has attributes a, b, and c, then you can use the following query:

SELECT a, b + c FROM class1;

FROM class1 is a very simple class expression (class_expression). Typically, class expressions can be complex constructs of basic classes, proxy classes, joins, and subqueries. However, you can also omit the class expression record and use the SELECT command as a calculator, such as SELECT 3 * 4. The expressions in the select_list (a,b+c in the above example) can also return varying results. For example, you can call a function with this method:

SELECT random();

The class_expression in the SELECT a, b + c FROM class1 example is very simple, containing only one class, class1. The class expression in the SELECT a, b + c FROM class1 example is very simple, containing only one class, class1. The class expression can actually be used with WHERE, GROUP BY, and HAVING clauses as needed. These clauses declare a series of class expressions for classes that originate from the FROM clause (i.e., the class to be queried), and the resulting class is used to generate the output, which is based on the select_list. Class expressions support complex transformations of classes, such as join classes. A join class is based on the rules of a specific join type Classes derived from two other classes, specifically TOTEM supports INNER, OUTER and CROSS JOIN connection types.

2.6.2 Cross-Class Query

To clearly describe cross-class queries, a detailed example is given here. Again, consider the cross-media data management scenario where a user wants to see to see other users' comments on a certain song, then first give the schema diagram for this scenario as follows:

music_lyr_TheBeatles_user1, music_lyr_TheBeatles_user2 are both proxy classes of music_lyr; music_lyr stores lyrics and songs, and its proxy class stores information such as user ratings, albums and lyrics.

If user1 wants to see user2's comments on the same songs belonging to the album "The Beatles", a simple cross-class query can achieve this, the specific SQL statement is as follows:

2.7 Update Object

The user can update an existing object with the UPDATE command. Suppose you find that all the thermometer counts for November 28 are two degrees lower, then you can update the data in the following way:

UPDATE weather SET temp_hi = temp_hi - 2, temp_lo = temp_lo - 2 WHERE date > '1994-11-28';

The new state of the object is shown below:

SELECT * FROM weather;

San Francisco | 41 | 55 | 0 | 1994-11-29
Hayward | 35 | 52 | | 1994-11-29
(3 rows)

2.8 Delete Object

The user can remove objects from the class with the DELETE command. Suppose you are no longer interested in Hayward's weather, then you can remove those objects from the class with the following method to remove those objects from the class:

DELETE FROM weather WHERE city = 'Hayward';

All weather records belonging to Hayward will be deleted.

SELECT * FROM weather;
city | temp_lo | temp_hi | prcp | date
---------------+---------+---------+------+------------
San Francisco | 46 | 50 | 0.25 | 1994-11-27
San Francisco | 41 | 55 | 0 | 1994-11-29
(2 rows)

We must be careful when using statements of the following form:

DELETE FROM classname;

If there is no condition, DELETE will remove all objects from the specified class, emptying it. The system will not ask you for confirmation until you do this!
